DEST-001 Fase 2 · Stage 1 (builder) Docker · cache de capas

El orden de las instrucciones decide la velocidad del build

Dockerfile · stage builder

La decisión

Copiar primero solo los manifiestos e instalar, y recién después copiar el resto del código:

# 1. solo package.json + package-lock.json
COPY package.json package-lock.json ./
RUN npm ci

# 2. recién ahora el resto del código fuente
COPY . .
RUN npm run build

Por qué — el mecanismo del cache de capas

Cada instrucción del Dockerfile (COPY, RUN…) produce una capa. En cada build, Docker reusa una capa cacheada si la instrucción y sus inputs no cambiaron. Para un COPY, el input es el contenido de los archivos copiados; para un RUN, el comando más el estado de las capas anteriores.

Regla clave: apenas una capa se invalida, todas las que siguen se reconstruyen. Como npm ci depende solo de package*.json (que cambia rara vez), su capa queda cacheada y la instalación de ~503 MB no se repite cuando solo tocás un .tsx.

Contraejemplo — el orden ingenuo

Si copiás todo el código antes de instalar:

COPY . .            # cualquier cambio de código invalida acá…
RUN npm ci          # …y reinstala los 503 MB en CADA build
RUN npm run build

Cada edición de una línea de código bustea el cache de npm ci. Builds lentos, y en CI eso se paga en cada push.

Para recordar

  • Ordená de lo que menos cambia a lo que más cambia. Dependencias (estables) arriba; código fuente (volátil) abajo.
  • Una capa invalidada arrastra a todas las siguientes → poné los pasos caros y estables lo más arriba posible.
  • Ángulo DevSecOps: builds rápidos y deterministas acortan el ciclo de CI, y un ciclo corto es lo que hace viable parchear seguido. La velocidad del build no es solo comodidad: es time-to-patch. Y npm ci (no install) mantiene la reproducibilidad con verificación de hashes, independiente del orden.
DEST-002 Fase 2 · Stage 1 (builder) Compilación · fuente vs artefacto

Tu app viaja compilada, no como src/ — y por qué migrate.ts necesita esbuild aparte

esbuild · next build · .next/standalone

El concepto

El código fuente (app/ + src/, en TypeScript) no es lo que corre en producción. Lo que corre es el artefacto compilado: next build transforma tu app a JavaScript plano y lo deja en .next/standalone. La carpeta src/ cruda no viaja a la imagen porque en runtime nadie la lee — el servidor lee el .next/ compilado.

app/ + src/ (lo que la app usa)  ──next build──▶  .next/standalone/   # app web
src/db/migrate.ts (entrypoint suelto) ──esbuild──▶  migrate.mjs        # migración

Analogía: en C escribís .c (fuente) y compilás a un binario; a producción mandás el binario, no los .c. El binario es tu programa, solo que no es la fuente legible.

Por qué migrate.ts necesita esbuild por separado

next build solo compila lo conectado al grafo de imports de la app web. Tu migrate.ts es un entrypoint suelto: ninguna página lo importa, lo ejecutás aparte con node (vía db:migrate). Por eso Next no lo ve y su versión compilada no queda en .next/standalone. esbuild compila ese único entrypoint huérfano → migrate.mjs: TS→JS y bundle de repository.ts/client.ts en un archivo, sin externos salvo los node: builtins.

La confusión común

«No veo src/ en la imagen, ¿falta mi app?» No: tu app está, compilada en .next/standalone (las páginas app/*.tsx aparecen como .next/server/app/*.js) y la migración en migrate.mjs. Lo único que no viaja es la fuente cruda; y el código muerto (p. ej. schema.ts, que nadie importa) no se compila a ningún lado.

Para recordar

  • Embarcá el artefacto, no la fuente. Fuente = receta; artefacto compilado = plato cocinado. En runtime solo corre el artefacto.
  • Los entrypoints fuera del grafo de la app necesitan su propia compilación. Un script CLI que el framework no conoce no aparece mágicamente en el build de la app.
  • Compilar en build time (esbuild/Next) y no en runtime (tsx) saca la maquinaria de compilación de la imagen → menos superficie de ataque.
  • Ángulo de seguridad: que la fuente no viaje es el principio correcto («solo lo que corre»); no es un control fuerte (el JS es reversible), pero reduce superficie y no expone la estructura legible del código.
DEST-003 Fase 3 · CI Supply chain · inventario

El lockfile de la app no es el inventario del artefacto

imagen base · undici en npm global · CVE-2026-12151

El concepto

El gate de Trivy frenó por CVE-2026-12151 (HIGH) en undici 6.25.0. npm why undici devolvió "No dependencies found" porque consultaba el árbol local de BudgetSpin, mientras que el log de Trivy ubicó el paquete en /usr/local/lib/node_modules/npm/node_modules/undici/package.json: lo incorpora el npm global de node:24-slim, fuera de package-lock.json.

Por qué — el artefacto hereda más que la aplicación

lo que DECLARÁS (package.json)  ⊂  lo que SHIPPEA (la imagen)

El artefacto también hereda SO, Node, npm y sus dependencias desde la imagen base. Por eso un lockfile limpio no prueba que la imagen esté limpia: hay que escanear e inventariar el artefacto final.

El impacto en la remediación

npm update undici u overrides en BudgetSpin no alcanzan un paquete gestionado por el npm global de la base. Las opciones reales son actualizar el digest de la imagen base, retirar npm del stage runtime si no se necesita, o aceptar el riesgo con alcance y vencimiento. En este caso se aceptó hasta 2026-09-22: producción ejecuta node directamente y el repo no usa new WebSocket, WebSocketStream, ws:// ni wss://, condiciones requeridas por el advisory.

Para recordar

  • Inventaria el artefacto, no solo el manifiesto. El SBOM de la imagen incluye paquetes heredados de la base que el lockfile de la app no conoce.
  • El SBOM del artefacto es lo más cercano a la verdad, pero no es omnisciente. Punto ciego: dependencias inlineadas/minificadas en un bundle sin package.json (el bundler borró el límite del paquete) — ni el SBOM las ve.
  • Antes de remediar, ubicá en qué capa vive la dependencia: app, framework, runtime o imagen base. Esa ubicación define qué actualización puede corregirla.
  • La aceptación vive en .trivyignore-gate y se aplica solo al gate; SARIF y JSON conservan el hallazgo para que aceptar no signifique dejar de verlo.
