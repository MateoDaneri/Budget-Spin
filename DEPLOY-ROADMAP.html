De «SSH y rezar» a un pipeline de producción real

Roadmap incremental para llevar el deploy de BudgetSpin desde el script manual actual hasta un pipeline CI/CD con artefactos inmutables, least privilege y observabilidad — usando exclusivamente herramientas gratuitas/OSS. Cada fase se implementa en una sesión, baja un concepto a tierra, y deja el sistema funcionando. El enfoque de seguridad va integrado en cada paso, no como sección aparte.

 Siguiente

Imagen reproducible y hardened

Superficie de ataque · reproducibilidad · least privilege en runtime

 Pendiente
  • next.config.ts: output: "standalone".
  • Dockerfile multi-stage: stage de build con npm ci; stage final solo con el output standalone.
  • Base Node 24 LTS slim pinneada por digest (no más tag flotante node:25-slim).
  • USER node: el contenedor deja de correr como root.
  • Resolver cómo corre db:migrate sin devDependencies en la imagen final.
  • El deploy no cambia: el servidor sigue haciendo build: — cero conflicto.
Ángulo de seguridad Single-stage como root con toolchain completa = máxima superficie de ataque. Multi-stage no-root con base pinneada = mínima. npm ci + digest pinning = integridad del artefacto.

Build & publish en CI

Build-once · artefacto inmutable · scanning como gate

 Pendiente
  • CI buildea la imagen tras los tests (buildx + cache de Actions).
  • Tags inmutables: ghcr.io/mateodaneri/budget-spin:sha-<short> + :latest.
  • Trivy escanea la imagen ANTES del push — gate en CRITICAL/HIGH.
  • syft genera SBOM como artifact del run.
  • Push a GHCR con GITHUB_TOKEN (packages: write).
  • Retención: limpiar tags viejos (verificar cuota de storage del free tier en docs de GitHub).
  • El servidor todavía no se toca: las imágenes quedan «en espera» en el registry.
Ángulo de seguridad La imagen que pasó tests y scan es bit a bit la que correrá en prod. SBOM = saber qué hay adentro cuando salga el próximo CVE de una dependencia.

4

CD: el deploy lo dispara CI

Acceso humano ≠ acceso de máquina · blast radius por credencial

 Pendiente
  • Servidor: usuario deploy dedicado + llave SSH propia restringida en authorized_keys con command="/opt/budgetspin/deploy.sh", no-pty, no-port-forwarding.
  • PAT fine-grained read:packages only en el server para docker login ghcr.io (solo puede LEER imágenes).
  • deploy.sh server-side: pull por tag → up -d → espera healthcheck → smoke test → exit code claro.
  • Compose: build: . → image: ghcr.io/...:${BUDGETSPIN_TAG}.
  • Job deploy en Actions con environment production y secret SSH_DEPLOY_KEY.
  • scripts/deploy-prod.sh local queda como break-glass (emergencias), en modo pull.
Ángulo de seguridad Si el secret de CI se filtra, el atacante solo puede disparar un deploy de lo que ya está en main — no abrir shell ni leer datos. Blast radius proporcional a la función de cada credencial.

5

Rollback instantáneo

Rollback como capacidad ensayada, no teoría · MTTR

 Pendiente
  • workflow_dispatch con input tag: re-deployar cualquier versión desde la UI de Actions.
  • Ensayo real: deployar sha-N, volver a sha-N−1, cronometrar.
  • Runbook: procedimiento paso a paso, incluido el caso «GitHub caído» (break-glass con imagen ya pulleada).
Concepto Un rollback que nunca se ensayó no existe. La pregunta de todo deploy: ¿cómo vuelvo atrás en 30 segundos?
Verificación Rollback ensayado y cronometrado, documentado en RUNBOOK.html.

Migraciones como paso explícito + backup

Migrar ≠ arrancar · backup antes de todo cambio irreversible

 Pendiente
  • Quitar db:migrate del command: del contenedor.
  • deploy.sh: backup timestampeado del sqlite (retención N) → migrar como paso separado → recién entonces up -d.
  • Documentar expand/contract (el porqué, aunque con 1 réplica no muerda).
Ángulo de seguridad Una migración fallida sin backup es pérdida de datos. El backup pre-deploy es la red de seguridad de la única parte del sistema que no se puede rebuildear: los datos.

Enterarse antes que el usuario · logs como evidencia

 Pendiente

Principios que gobiernan el roadmap

  • Build once, deploy everywhere: el artefacto se construye una vez en CI; prod nunca compila.
  • Sin gastar: ante alternativas, siempre la OSS/free si el objetivo de aprendizaje es el mismo.
  • Orden anti-conflictos: hardening de imagen (F2) antes de publicarla (F3); el compose cambia a image: recién con el CD (F4); migraciones (F6) después de que exista el deploy script definitivo.
  • Una fase por sesión, sistema funcionando al final de cada una, RUNBOOK.html y este roadmap actualizados en el mismo turno.
  • Seguridad integrada: cada fase explicita su ángulo (supply chain, superficie de ataque, least privilege, integridad, evidencia).
