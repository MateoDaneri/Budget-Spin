OBS-001 Fase 1 · CI Integridad del entorno Resuelto

Árbol de node_modules corrupto: tipos fantasma y Permission denied

2026-06-09 · node_modules/

Contexto

Preparando el entorno local para correr typecheck y tests, antes de armar el pipeline de CI.

Síntoma

  • rm -rf node_modules fallaba con Permission denied.
  • El typecheck tiraba Cannot find type definition file for 'chai 2' (y similares) — tipos que no existían en ninguna dependencia declarada.

Evidencia — verificado, no asumido

  • Aparecían carpetas duplicadas con sufijo  2 (@types/chai 2, etc.) dentro de node_modules — la firma típica de un install interrumpido o concurrente.
  • Algunos directorios tenían permisos no-propios, lo que bloqueaba el borrado del usuario normal.

Causa raíz

Instalación de npm corrupta: directorios duplicados y permisos inconsistentes. TypeScript levantaba los @types/* 2 como archivos de definición válidos y se quejaba de que el paquete real (chai) no estaba — eran definiciones fantasma, no un problema del código.

Comportamiento esperado

Un árbol de dependencias limpio y reproducible, reconstruido exactamente desde el lockfile.

Resolución

sudo rm -rf node_modules && npm ci

npm ci (no install) borra el árbol y lo reconstruye bit a bit desde package-lock.json, sin re-resolver rangos.

Lección para estudiar

  • Ante errores de tipos fantasma o módulos que "no existen", sospechar del árbol de node_modules antes que del código.
  • Las carpetas con sufijo  2 son señal de un install corrupto/concurrente.
  • npm ci da un árbol determinista; install no garantiza lo mismo.
  • Seguridad: un install no determinista es también un riesgo de integridad — no sabés con certeza qué quedó en el árbol que vas a ejecutar.
OBS-002 Fase 1 · CI Clase de bug: timezone Resuelto

OOM en los tests: un bug de timezone en addMonths que disparaba un loop infinito

2026-06-09 · src/finance/dates.ts · commit 09405ff

Contexto

Corriendo la suite de Vitest en local para dejarla verde antes de meterla como gate en CI.

Síntoma

FATAL ERROR: JavaScript heap out of memory al correr repository.test.ts. El proceso de Node se comía toda la RAM y moría.

Evidencia — el síntoma estaba lejos de la causa

  • Aislando tests con -t y corriendo con --reporter=verbose, el grep del crash header reveló que el test que crasheaba era el #10 ("tracks plan items") — que llama a calculatePlanProjection — y no el #6 que el output verbose sugería al principio.
  • El run compuesto ocultaba cuál era el test real; recién aislado quedó claro.

Causa raíz

addMonths construía la fecha con Date.UTC(...) pero la leía con getters de hora local. En UTC−3 el resultado era un no-op: devolvía el mismo mes que recibía. Eso hacía que monthsInclusive nunca llegara al mes final → loop infinito acumulando elementos → OOM.

Comportamiento esperado

addMonths debe ser independiente de la timezone del host: sumar N meses da el mismo resultado en UTC, en UTC−3 o donde sea.

Resolución — dos capas

El bugAplicado

Construir y leer la fecha consistentemente en UTC (Date.UTC + getters UTC). Tests de regresión en tests/dates.test.ts que afirman independencia de timezone.

El enmascaramientoAplicado

En ci.yml: TZ: America/Montevideo. Los runners corren en UTC por default, lo que habría dejado pasar este bug — forzar una TZ no-UTC en CI lo atrapa.

Lección para estudiar

  • Los runners en UTC esconden toda una clase de bugs de timezone. Verde en CI ≠ correcto en tu zona; correr al menos un job con TZ no-UTC.
  • El síntoma (OOM) puede estar a kilómetros de la causa (aritmética de fechas). Seguir la cadena, no el primer error.
  • Fechar/timestampear los veredictos antes de confiar: el reporte verbose apuntaba al test equivocado.
  • Los tests aislados exponen lo que el run compuesto oculta.
OBS-003 Fase 1 · CI Supply chain Resuelto

Pinnear una Action por SHA: el tag anotado no es el commit que corre

2026-06-10 · .github/workflows/ci.yml

Contexto

Pinneando las Actions por SHA de commit (en vez de tags mutables) para cerrar el vector de supply chain del estilo tj-actions/changed-files (2025): un repo de Action comprometido puede mover un tag a código malicioso.

Síntoma

El SHA "obvio" que aparecía para el tag v6.0.3 de actions/checkout no era el commit que realmente ejecuta la Action.

Evidencia

v6.0.3 es un tag anotado: apunta a un objeto tag con su propio hash (9f69817…), no directamente a un commit. El commit dereferenciado — el que corre — es otro (df4cb1c…).

Causa raíz

En git, un tag anotado es un objeto propio con su propio SHA. Lo que tiene que ejecutar la Action es el commit al que el tag apunta, no el objeto tag. Confundirlos pinnea a un objeto que no es código ejecutable.

Resolución

Dereferenciar el tag al commit y pinnear ese. En ci.yml quedó:

uses: actions/checkout@df4cb1c069e1874edd31b4311f1884172cec0e10 # v6.0.3

El comentario con el tag legible es para humanos; el SHA es la verdad inmutable.

Lección para estudiar

  • Pinnear por SHA es la defensa contra el repunte malicioso de tags — pero hay que pinnear el commit correcto.
  • Tag anotado vs lightweight es una distinción que muerde justo acá: solo el anotado tiene un hash propio distinto del commit.
  • El SHA es content-addressed: no se puede repuntar. El tag es un puntero mutable.
OBS-004 Fase 1 · CI Least privilege Resuelto

Gitleaks devolvía 403 en eventos pull_request

2026-06-10 · job secrets-scan en ci.yml

Contexto

El workflow declaraba permissions: contents: read a nivel global (least privilege: zerea todos los scopes no mencionados). En el primer run sobre un evento pull_request, el job de gitleaks falló.

Síntoma

gitleaks-action devolvía 403 al intentar listar los commits del PR vía API.

Evidencia — la respuesta dice el permiso exacto

Leyendo los headers de la respuesta de GitHub:

x-accepted-github-permissions: pull_requests=read

El error no era ambiguo: nombra el permiso que falta.

Causa raíz

Declarar permissions: contents: read a nivel workflow pone todos los demás scopes en none, incluido pull-requests. En eventos pull_request gitleaks lista los commits del PR por API → necesita pull-requests: read, que estaba en none.

Comportamiento esperado

Cada job con exactamente los permisos que usa: ni más (superficie innecesaria) ni menos (rompe).

Resolución

Override a nivel de job (no de workflow) solo en secrets-scan:

permissions:
  contents: read
  pull-requests: read

Los otros dos jobs mantienen el default estricto. Además GITLEAKS_ENABLE_COMMENTS: "false": comentar en el PR necesitaría pull-requests: write, y en runs de Dependabot el token está capado a read igual. La señal es el job en rojo; el detalle vive en el log.

Lección para estudiar

  • Declarar permissions: zerea lo no mencionado — ese es el objetivo (least privilege), pero hay que conceder explícitamente lo que cada job realmente usa.
  • El override a nivel de job mantiene el principio sin abrir todo el workflow.
  • Los headers de error de una API bien diseñada te dan el permiso exacto: leer la evidencia en vez de adivinar y abrir scopes de más.
OBS-005 Fase 1 · CI Control de cambios Resuelto (con decisión)

La branch protection no se enforzaba en un repo privado del plan Free

2026-06-11 · GitHub Ruleset protect-main

Contexto

Configurando la protección de main vía Ruleset: requerir PR, requerir los 3 checks de CI, sin bypass.

Síntoma

El ruleset se creaba, pero con un warning de que no se enforzaría en este repositorio.

Evidencia

El aviso de la UI al crear el ruleset, confirmado contra la documentación: rulesets (y branch protection clásica) en repos privados requieren plan Team/Enterprise. En GitHub Free, los rulesets solo enforzan en repos públicos.

Causa raíz

Limitación de plan, no de configuración: GitHub Free no enforza protección de branch en repos privados. El ruleset existía pero era decorativo.

Comportamiento esperado

Que el push directo a main y el merge sin checks estén físicamente bloqueados, no dependientes de disciplina.

Resolución — la decisión

Hacer el repo público, tras auditar que el historial estuviera limpio (sin secrets — apoyado en gitleaks de OBS-004 y la higiene de .gitignore de Fase 0, que dejó los .sqlite fuera del repo). Beneficio extra: el code scanning con SARIF gratis es solo para repos públicos.

Lección para estudiar

  • Algunas features de server-side enforcement están detrás de un paywall: conocer el límite del plan antes de diseñar el control.
  • Antes de hacer público un repo, auditar el historial: lo commiteado es para siempre (clones, reflogs, cachés).
  • Seguridad: público amplía la superficie de exposición del código — la verificación de "no hay secrets" va ANTES de publicar, no después. Acá fue aceptable porque los datos (.sqlite) nunca entraron al repo.
OBS-006 Fase 1 · CI Mantenimiento de deps Resuelto

Dependabot: conflictos seriales de lockfile en cascada

2026-06-11 · .github/dependabot.yml · PR #7

Contexto

Dependabot abrió varios PRs de actualización de dependencias a la vez.

Síntoma

Al mergear uno, los otros empezaban a dar conflicto y pasaban de ✓ a ✗.

Evidencia

Cada PR individual edita package-lock.json. Tras mergear el primero, los demás quedaban desactualizados respecto de main → conflicto en el lockfile. El modo estricto ("require up to date") del ruleset lo amplificaba: cada uno tenía que re-actualizarse contra el nuevo main.

Causa raíz

Problema estructural N²: N PRs que tocan el mismo archivo compartido (el lockfile) generan conflictos en cascada cuando se mergean de a uno.

Resolución — arquitectónica, no procedimental

Agregar groups a dependabot.yml: un grupo npm-minor-patch (todas las minor/patch en UN PR) y un grupo actions. Un PR, un cambio de lockfile, cero conflicto serial.

groups:
  npm-minor-patch:
    applies-to: version-updates
    patterns: ["*"]
    update-types: ["minor", "patch"]

Lección para estudiar

  • Cuando varios cambios automáticos tocan un archivo compartido, agruparlos resuelve el problema en la arquitectura. Mergear rápido o rebasear a mano es un parche que no escala.
  • El modo estricto del ruleset es correcto pero amplifica este patrón → la respuesta es agrupar, no relajar el control.
OBS-007 Fase 1 · CI Mantenimiento de deps Decisión consciente: no mergear

ESLint 10 incompatible con eslint-config-next

2026-06-11 · PR #5 (dejado abierto)

Contexto

Dependabot propuso subir ESLint a la versión mayor 10.

Síntoma

El PR de ESLint 10 quedaba en rojo: el step de lint rompía con la config de Next.

Causa raíz

eslint-config-next todavía no soporta ESLint 10 — incompatibilidad de versión mayor entre un peer y el nuevo major.

Resolución — la decisión correcta es no mergear

Dejar el PR #5 abierto como recordatorio vivo de que el upgrade está pendiente de soporte upstream. Mismo criterio con TypeScript 6 (PR #10) y @types/node 25: majors que se postergan hasta que el ecosistema acompañe.

Lección para estudiar

  • No todo lo que Dependabot propone se mergea: los majors pueden romper peers.
  • Dejar el PR abierto es un recordatorio barato y visible; la decisión "esperar a upstream" queda documentada en vez de perderse.
  • Seguridad: hay una tensión entre estar al día (parches) y estabilidad — un major apurado puede romper más de lo que arregla.
OBS-008 Fase 2 · Imagen hardened Supply chain / fuga de datos Resuelto (capas A + B aplicadas)

El trace de standalone filtró la base de datos de desarrollo al artefacto

2026-06-13 · next.config.ts, src/db/client.ts

Contexto

Activamos output: "standalone" en next.config.ts, el primer paso de la Fase 2. Standalone hace un file tracing: analiza estáticamente el código compilado y copia a .next/standalone/ solo los archivos que la app alcanza en runtime (pasó de 319 paquetes / 503 MB a 12 / 38 MB).

Síntoma

Dentro del output apareció algo que no era código: .next/standalone/data/budgetspin.sqlite — la base de datos de desarrollo, con datos sembrados.

Evidencia — verificado, no asumido

  • Los archivos de trace .nft.json de casi todas las rutas listan data/budgetspin.sqlite como dependencia: nft la decidió incluir, no fue ruido.
  • mtime: la copia en standalone/ tiene fecha del build; el original ./data/ es viejo → fue una copia hecha durante el build.
  • Contenido: la sqlite copiada tiene las tablas sembradas (users, currencies, categories, income_items…) → es la DB de dev real, no un archivo vacío.

Causa raíz

El tracer (@vercel/nft) hace análisis estático de las operaciones de filesystem. Vio la ruta literal de src/db/client.ts:

path.join(process.cwd(), "data", "budgetspin.sqlite")

y aplicó su heurística: "el código lee este archivo → es un asset que tiene que viajar con la app". El problema: nft no puede distinguir un asset estático a hornear (un config.json, un .wasm) de estado mutable de runtime (una base de datos). En el código las dos se ven igual: fs.read(path).

Comportamiento esperado

La imagen debe ser inmutable y stateless: solo código. La base de datos no existe en el artefacto — vive en el volume de Docker (budgetspin-prod-data:/app/data), la crea db:migrate en el primer arranque, y persiste entre deploys. El data/ de la imagen debería estar vacío o no existir.

Por qué .dockerignore no alcanzaba

Ya excluía data/*.sqlite, pero el .dockerignore solo afecta el build context (lo que ve un COPY . .). En un Dockerfile multi-stage, el stage final copia con COPY --from=builder entre stages, donde la sqlite ya vive dentro de standalone. Es mitigación, no fix.

Resolución — dos capas (causa raíz + defensa en profundidad)

Capa A — la fuenteAplicado

En next.config.ts: outputFileTracingExcludes: { "*": ["data/**"] }. Le dice a nft que no bundlee data/**. Corrige en el origen y sirve aunque se buildee fuera de Docker. Verificado: tras rebuild, .next/standalone/data desaparece.

Capa B — la imagenAplicado

En el Dockerfile: rm -rf .next/standalone/data en el stage builder. Red de seguridad determinista por si A falla, cambia la versión de Next, o el build crea una DB durante "Collecting page data".

Lección para estudiar

  • Las herramientas de build que "ayudan" copiando assets pueden arrastrar estado mutable. Lo que para vos es una DB, para el tracer es "un archivo que el código lee".
  • Distinguir asset de build vs. estado de runtime es un principio central de contenedores inmutables: el estado va en volumes/servicios externos, nunca horneado en la imagen.
  • Defensa en profundidad: corregir en la fuente (A) y poner una red determinista (B); cada capa puede fallar sola.
  • El .dockerignore no cubre lo que pasa entre stages de un multi-stage build.
  • Verificar, no asumir: mtime + contenido + los .nft.json confirmaron el mecanismo antes de afirmar la causa.
